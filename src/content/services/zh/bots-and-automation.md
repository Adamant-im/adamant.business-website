机器人是能动用资金的小程序。这一认知驱动我们每一个决策：最小化密钥权限、追求稳定可靠、完整源码交付给你。我们交付软件——密钥、策略与责任由你掌控。

## 我们自动化的内容

- **通知与告警机器人** — 通过 Telegram 与 ADAMANT 监控余额、节点健康、交易所订单状态或链上转账，在问题演变为损失前通知人工介入
- **执行辅助工具** — 半自动化流程：机器人准备交易或订单，由人工审批；适用于资金运营与 OTC 结算
- **运营仪表盘** — 将团队已在运行的钱包、节点与机器人集中在一个视图，告别八个浏览器标签页加一张表格
- **聊天内服务** — 运行在端到端加密 ADAMANT 聊天中的机器人：兑换流程、付款、支持队列与访问控制
- **AI 辅助工具** — 基于 LLM 的摘要与分诊助手，始终经资深工程师审核，且绝不直接接触密钥

## 如何避免机器人变成事故

大多数机器人事故源于同样的三个错误：具备提现权限的 API 密钥、导致重复支出的重试循环，以及故障时毫无声息。我们的标准做法：

- 交易所密钥仅授予工作流所需的最小权限——尽可能仅限交易或只读
- 幂等操作与显式状态，确保重启不会重复执行转账
- 为每个外部 API 设置速率限制预算与熔断机制
- 对机器人本身进行告警——静默停止的机器人比没有机器人更危险
- 部署在你的服务器上，日志与指标由你的团队可读可查

## 案例：链上验证的 2FA

[adamant-2fa](https://github.com/Adamant-im/adamant-2fa) 通过 ADAMANT 即时通讯传递一次性验证码，而非短信。投递在链上验证，无法被 SIM 卡劫持。我们为邮件与短信不可接受为攻击面的产品，构建类似的验证流程。

## 案例：加密聊天内的交易所

[adamant-exchangebot](https://github.com/Adamant-im/adamant-exchangebot) 在端到端加密聊天中运行完整兑换流程——报价、充值、付款。同一模式可适配付款、水龙头、赏金发放，以及你自有技术栈中的内部代币分发。
